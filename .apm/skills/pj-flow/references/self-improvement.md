# self-improvement — pj-flow スキル自体を育てる運用

このスキルはグローバルに1個。runner のスキル配置にある `pj-flow/` を直接 Edit することで、**どのリポジトリのどのスレッドからでも、同じスキルを改善できる**。

## 起動条件（ユーザーが言ったときに自動発動）

- 「pj-flow のここで止まらないで欲しい」
- 「pbcopyのタイミング遅い」
- 「テンプレに○○欄が欲しい」
- 「Migration の手順がわかりにくい」
- 「○○のシナリオが抜けている」
- スキルの使い勝手・抜け漏れ・冗長性に関する任意のフィードバック

## 手順

### 1. 不満の本質を1行で復唱

```
「○○のときに△△している、これを□□に変える」という理解で合っているか
```

復唱で **対象（どの挙動）**と **変更後の動き**を明確化する。AskUserQuestion でも可。

### 2. 該当箇所を特定

- `SKILL.md` のどのセクションか
- `templates/<...>/<...>.md` のどのファイルか
- `references/<...>.md` のどこか

特定できないならユーザーに「どこに書かれているはず」を聞く。**スキルが大きくなったときの索引は SKILL.md §9**。

### 3. Edit で恒常化

- SKILL.md / templates / references を直接 Edit
- 旧文言は **コメントで残さない**（後から読みづらくなる）。Git履歴に任せる
- 大きな構造変更（節を新設・削除）の場合は SKILL.md §9 の索引も更新

### 4. 変更点をユーザーに報告

- どのファイルの、どこを、どう変えたか（1〜3行）
- 副作用（他リポでも挙動が変わる）の明示
- 必要なら commit を促す（スキル配置ディレクトリが git 管理されている場合）

## やってはいけない自己改善

- **対症療法的な「if 文の追加」**: 「特定ユーザーが Xと言ったら Yする」式の特例追加は禁止。**原則レベルで書き換える**
- **古い文言の温存**: 「旧」「new」みたいなセクション併存は避ける。新規約に統一する
- **テンプレに「TODO」を埋める**: テンプレは `cp -r` で使うもの。プレースホルダ以外の `TODO` は混ぜない
- **トリガーフレーズを SKILL.md description で**サイレントに削る: ユーザーが使っていたフレーズを削るとスキルが起動しなくなる。削るならユーザーに確認

## このスキルの版管理

スキル配置ディレクトリの版管理方式は環境ごとに異なる。次の順で確認する:

- APM / Git repo が source of truth なら、そこで commit / push する
- dotfiles 管理なら、dotfiles 側の規約に従って取り込む
- どちらでもなければ、変更点を最終報告で明示し、ユーザーに永続化方針を確認する

## このスキルの命名・移動

`pj-flow` という名前を変えたくなった場合:
1. runner のスキル配置で `pj-flow/` を `<new-name>/` に rename
2. 複数 runtime に symlink している場合はすべて張り直し
3. SKILL.md description のスキル名表記を全置換
4. references のスキルパス記述を全置換

## このスキルの destroy

スキル配置から `pj-flow/` を消すと、各リポジトリの `<PROJECTS_ROOT>` 配下は**残るが新規Threadの初期化が止まる**。
destroy する場合は:
1. テンプレを再利用するなら別の場所（例: `~/dotfiles/templates/`）に移してから消す
2. 各リポジトリの `<PROJECTS_ROOT>` は**自走できる**（規約は OBJECTIVE.md / OUTPUT.md に内蔵されている）ので、即時の影響は新規Threadだけ
